Big O and Other Algorithm Notation

An algorithm in computer science is a set of steps for a computer program to accomplish a task. Algorithms put the 'Science' in Computer 'Science'. Across all sciences, efficient algorithms are needed to analyze gigantic data sets or select the right choice intelligently.

There are many different types of algorithms but not all algorithms work well in every situation. The pricipal criterion for a good algorithm are:
1) It must be solve the problem
2) AND it must do so efficiently

Now that we know what algorithms are supposed to do, it's important to understand why most of them can't do their job. The primary purpose for an algorithm is to solve a problem. Well, it turns out most algorithms can't be solved correctly, why? It can take a really, really, really long time. Example:

Lets say we wanted to write a program that finds out the most efficient route for you to meet all your friends in NYC. The route starts and ends at your house.
-You have 20 friends
-They are all stationary

Calculation:
19! (factorial of 19) = 121,645,100,408,832,000

It could take a week or more to run this. Instead, most programmers would settle with writing a program that's "ok", but solves the problem in seconds. In most cases "ok" is good enough.

How Efficiency is Measured

The obvious metric would be to time how long the code takes to run from start to finish. Unfortunately that metric is corrupted because it only describes the run-speed on that particular input, programming language and computer-hardware.

Enter Asymptotic Analysis! It is a system of notation that computer scientists use to describe how long an algorithm will take to solve a problem, independent of hardware and programming language. That way we can come to a conclusion that there are some algorithms that are better than others.

Big O Notation

Big 'O' is the representation of how fast a program's runtime grows asymptotically. It measures the runtime of your program as the size of your inputs grows towards infinity. e.g. How does it sort an array of size 10 versus size 100,000?

O(n)

Let's say we wanted to count the number of characters in a string in the most basic way. Iterate over the string, letter by letter adding 1 to a counter variable. 

            
              var = "mouse";

              for(var count = 1; count <= mouse.length; count++) {
                console.log(count);
              }
              //1, 2, 3, 4, 5

This algorithm runs in Linear Time with respect to the number of characters in the string, 'n' because (n= 5). It is said to have O(n) notation because the time used to traverse the entire string is proportional to the number of characters. Each character takes the same amount of time to look at and the run time increases linearly with the input length.

O(1)

Perhaps you are creating huge strings and want a faster method of retrieving the value. You decide store the value in an array. Now, you can call '.push()' or '.pop()' to add or retrieve a value inside the array. 'push' and 'pop' are said to take asymptotically constant time. Why?

Think back to what Big O measures. How does your program's runtime change as your input approaches infinity? In this case it wouldn't matter if your string was 2 characters or 2000 characters, 'push' or 'pop' would accomplish the desired task at the same speed. That's why this example would be considered to be running in constant time.

O(log n)

Lets say you were searching for a particular number inside an array of sorted numbers. For example:

The algorithm is searching for number 3 in this array. [1, 2, 3, 4, 5, 6, 7]

-The algorithm starts at the halfway point, in this case 4 and asks, "Is the element I want greater, less than, or equal to this?"

-It is less than so all the numbers that are greater than 4 are cut out and we're left with: [1, 2, 3]

-The algorithm starts at the halfway point again and does the same with 2. We are left with 3.

This algorithm is known as the "binary search" algorithm and cuts the size of the array in half with each operation. An array of size 8 will have at most, 3 operations. An array with size 16 will have at most, 4 operations. That means doubling the size of the array will only increase the run time by one chunk of code. It's said to run in logarithmic time. (log2 8) = 4 because 2x2x2 = 8
